Hvordan er 3D og 2D koblet sammen?
De to visninger er tæt koblet via det samme datagrundlag – der er ingen duplikering af logik. Kernen er en 2D “blueprint” (et simpelt grid af tegn) plus en farve-map, som begge renderingsmetoder trækker på.
Fælles data-model (findes i gem-typerne):

gem.data: Et 2D-grid (array af strings eller lignende) – det er den “blueprint”, der definerer formen af ædelstenen/jewelryen.
colorMap / resolveColor-logik: Bestemmer hvilke farver hver karakter i grid’et skal have.
Andre egenskaber (palet, renhed, mønstre, metaller osv.) fra generate.ts og types.ts.

Denne data sendes videre til begge visninger.
2D-visning i menuen (bruges i inventory, cards, shop osv.):

Hovedfil: src/gem/draw2d.ts → funktionen drawGem() (eller lignende drawJewelry2d).
Tegner direkte på et HTML Canvas (2D context).
Bruges i komponenter som PixelItemCard.tsx, GemCard.tsx eller tilsvarende menu-komponenter.
Resultat: En flad, pixel-agtig 2D-ikon/visning, der er hurtig og skalerbar til UI.

3D-versionen (i scenen / viewer):

Hovedfil: src/gem/drawJewelry3d.ts (eller tilsvarende i drawing.ts).
Tager præcis samme 2D-blueprint og “ekstruderer” den til 3D:
Hvert lag i grid’et bliver til en voxel-lag.
Bygger en Voxel3DGrid (eller lignende struktur).
Tilføjer dybde, metaller, gems i slots osv.

Bruges i GemViewer.tsx, VoxelScene.tsx og React Three Fiber-komponenterne (@react-three/fiber).
Resultat: Den fulde 3D-voxel-model du ser i spillet.

Flowet i praksis:

En gem/jewelry genereres → får en data-blueprint + farver.
Menuen kalder drawGem() fra draw2d.ts → viser 2D-ikon i kort/liste.
Når du klikker/åbner viewer, kalder den drawJewelry3d() → samme blueprint bliver til 3D-mesh.
Alt er stateless og deterministisk: Samme input → samme output i både 2D og 3D.

Det er en meget elegant og genbrugelig arkitektur – ingen dobbeltarbejde, og det gør det nemt at tilføje nye mønstre eller effekter (ændring i blueprint eller colorMap påvirker begge visninger med det samme).
Hvorfor det føles “koblet på en måde”
Fordi det er det samme grunddata. Der er ikke to separate modeller – det er én blueprint, der bare tolkes forskelligt:

2D = flad tegning på canvas.
3D = voxel-ekstrudering + Three.js mesh.

Hvis du har en specifik gem/jewelry, der opfører sig mærkeligt mellem menu og 3D, så er det sandsynligvis et sted i drawJewelry3d.ts eller i hvordan VoxelScene håndterer slots/lag.

Forslag til ren afkobling (uden at ødelægge noget andet)
Tilføj et valgfrit override-felt i din eksisterende type-definition (f.eks. i src/gem/types.ts). Det påvirker kun sværd og hakket:
TypeScript// Tilføj dette til din Item / Jewelry / Gem type
customVisuals?: {
  menu2D: string;      // f.eks. "/assets/weapons/svaerd_2d.png"
  scene3D: string;     // f.eks. "/assets/weapons/svaerd.glb"
  scale?: number;      // valgfri justering
  rotation?: [number, number, number];
  positionOffset?: [number, number, number];
};
Hvordan du implementerer det (minimalt):

Opret mappe: public/assets/weapons/ (eller src/assets/weapons hvis du vil importere).
Når du genererer/crafter sværd eller hakket (i crafting.ts eller en ny weapons.ts), sæt feltet:TypeScriptconst sword = {
  ...baseData,
  name: "Flammende Sværd",
  customVisuals: {
    menu2D: "/assets/weapons/svaerd_2d.png",
    scene3D: "/assets/weapons/svaerd.glb",
    scale: 1.2
  }
};
Opdater 2D-menu-logikken (f.eks. i din PixelItemCard.tsx eller drawGem()-wrapper):TypeScriptif (item.customVisuals?.menu2D) {
  return <img src={item.customVisuals.menu2D} alt={item.name} className="..." />;
} else {
  return drawGem2D(item); // gammel kode – virker stadig for gems
}
Opdater 3D-scenen (f.eks. i GemViewer.tsx eller VoxelScene.tsx):TypeScriptif (item.customVisuals?.scene3D) {
  const { scene } = useGLTF(item.customVisuals.scene3D); // fra @react-three/drei
  return <primitive object={scene} scale={item.customVisuals.scale || 1} />;
} else {
  return <ProceduralVoxelGem item={item} />; // din eksisterende voxel-kode
}
Tilføj useGLTF (fra @react-three/drei) hvis du ikke har det allerede – det er én linje i package.json.
